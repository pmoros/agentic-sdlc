# Agent Instructions

This repository is the **Agentic SDLC toolbox**: scripts, runbooks, and the
`.agents/` commands/rules/skills used to run work sessions. It holds no
session state itself — not even gitignored. All session state (backlog,
active/paused/done sessions, plans, decisions, worktrees) lives in the
sibling `work-sessions` repo, which commits everything so it survives across
machines and time.

---

## Integration Schemas

**None of the files below auto-load.** Read the relevant file **before any
operation** with that system — this is the loading mechanism, not a
suggestion (see `02-adrs/0001-tiered-conditional-rule-loading.md`). Each file
also self-declares its own trigger in a note near the top.

| System | Schema file | Load when... |
|---|---|---|
| Atlassian — Jira / Confluence | `.agents/rules/atlassian.instructions.md` | before any Jira/Confluence read or write |
| AWS | `.agents/rules/aws.instructions.md` | before any AWS CLI/MCP call or IaC (CDK/Terraform) operation |
| GitHub | `.agents/rules/github.instructions.md` | before any PR/branch/commit-adjacent GitHub operation |
| Production deployments (risk/pre-flight/rollback) | `.agents/rules/deployments.instructions.md` | before authoring or executing a deployment definition |
| Development lifecycle (staged pipeline + Gate A/B) | `.agents/rules/dev-lifecycle.instructions.md` | before design, implementation, QA, or a gate review |
| Session state maintenance (keep tracking files live) | `.agents/rules/session-state.instructions.md` | at session start, and at pause/resume/stop/end |
| Engineering doctrine (TDD/BDD/design-first/KIS) | `.agents/rules/engineering.instructions.md` | before design or implementation — the compact reminder in Framework Rules below covers routine sessions |

The Atlassian and AWS rule files ship as **templates** with placeholder
projects/values — fill in your org's actual projects, fields, and
conventions before relying on them. Add further integration schemas here as
your org adopts more systems (e.g. a monorepo build tool's gotchas, a
different cloud provider, a different ticketing system).

> `.github/instructions/` contains symlinks to these files for GitHub-native
> tooling compatibility. Where a rule's trigger correlates with a file type
> (AWS/IaC), the symlinked copy carries `applyTo:` frontmatter so GitHub
> Copilot scopes it natively. Operation-triggered rules (Jira, GitHub) have
> no reliable Copilot file-glob equivalent — Copilot users open the file
> manually before that operation. See ADR-0001 for the full rationale.

### Jira Execution Rule In This Manager Repo

When Jira ticket work is initiated from this SDLC toolbox repo, use `.agents/rules/atlassian.instructions.md` as the operational source of truth for ticket creation, updates, transitions, comments, and field handling.

This applies to Jira work generally from this repo, for any project documented in the Atlassian instruction file. If a ticket must be created and the field contract is already known, create it directly instead of detouring through subagents, broad tool discovery, or generic search.

When Jira MCP tools are exposed in the session, use the discovered Jira MCP tool for the required operation directly (`create`, `get`, `edit`). Do not re-discover tools repeatedly once a valid operation tool is already available, and do not use CLI/REST fallback before the discovered Jira MCP operation for that task fails.

If a reusable Jira gap is discovered while working from this repo:
- update the repo instruction file when the gap is a stable process rule
- or store it in repo-scoped memory when it is a workspace-level reminder

Do **not** promote ticket-specific, worktree-specific, or session-specific Jira behavior into global memory.

### AWS / CDK Execution Rule In This Manager Repo

When running local CDK (or other IaC) operations from this SDLC toolbox repo — especially plan/diff/synth or apply/deploy from a target repo worktree — consult `.agents/rules/aws.instructions.md` first.

Treat environment-selecting inputs (e.g. `AWS_PROFILE`, an environment ID, a branch/version selector) as deployment inputs, not defaults. If a change is expected to be an in-place update but the diff shows full resource creation, re-check the AWS instruction file before proceeding.

## Repo Doctrine

## Session State

All session state lives in the sibling `work-sessions` repo —
backlog/wip/scratchpad (`work/`), the session registry (`SESSIONS_STATE.md`),
and per-session folders (`sessions/<id>/CONTEXT.md`, `PLAN.md`, `SPEC.md`,
`TASKS.md`, `WORKLOG.md`, `adrs/`, `docs/`, `journal/`, `scripts/`, `tmp/`,
`worktrees/`).
See that repo's `README.md` for the full schema and workflow.

This repo (`agentic-sdlc`) never holds session state itself.
The only thing it contributes to a session is read-only tool access: every
session gets a worktree of this repo at `worktrees/agentic-sdlc/` (detached,
on the default branch, kept in sync) — created by
`#initialize_work_session_folder.prompt.md` and refreshed by
`#resume_work_session.prompt.md`.

### Reference Repo Policy

Every repo under `repos/` — including this one — is a **read-only source of
truth**: always checked out on its default branch, always kept in sync with
`origin`, and **never edited directly**. All real work happens only in a
worktree created under a session's `worktrees/` folder in `work-sessions`.
`scripts/create-worktree.sh` enforces this by syncing the source repo before
creating anything, and refusing to proceed if it has local changes rather
than discarding them. See `docs/create-worktree.md`.

There is no branch-only/no-worktree escape hatch — `#create_work_tree.prompt.md`
is the only way a target repo gets a place for work to happen.

### Session lifecycle

Use these in order: `start-work-session` (skill) to begin →
`#pause_work_session` / `#resume_work_session` to save/reactivate state
without closing → `#stop_work_session` for a hard stop that keeps branches →
`#end_work_session` to close out completed work. See the Skills Reference
table below for what each does.

## Local MCP Setup (per contributor)

`.vscode/mcp.json` is **gitignored** — each contributor must create their own local copy before MCP-dependent features (Jira, Confluence, AWS) will work.

### First-time setup

1. Copy the example file:
   ```
   cp .vscode/mcp.example.json .vscode/mcp.json
   ```
2. Restart VS Code — it will **prompt you for credentials** on first use and store secrets in the system keychain.

The file uses VS Code's `inputs` feature — no secrets are ever hardcoded or written to disk.

VS Code will prompt for the following on first use:

| Input | What to provide |
|---|---|
| Atlassian Jira URL | e.g. `https://yourcompany.atlassian.net` |
| Atlassian Confluence URL | e.g. `https://yourcompany.atlassian.net/wiki` |
| Atlassian account email | Your Atlassian login email |
| Atlassian API token | A **classic** API token from [id.atlassian.com/manage-profile/security/api-tokens](https://id.atlassian.com/manage-profile/security/api-tokens) |
| AWS profile | Your AWS CLI profile name (e.g. your SSO profile) |
| AWS region | e.g. `us-east-1` |

### Agent diagnostics — run this check at the start of any session

If any MCP-backed operation fails or the user reports MCP tools are unavailable:

1. **Check for `mcp.json`**:
   ```
   test -f .vscode/mcp.json && echo "EXISTS" || echo "MISSING"
   ```
   If `MISSING`, tell the user:
   > "`mcp.json` is missing from `.vscode/`. Copy it from the example: `cp .vscode/mcp.example.json .vscode/mcp.json`, then restart VS Code — it will prompt for credentials on first use."

2. **Check `uvx` is on PATH**:
   ```
   which uvx || echo "uvx not found"
   ```
   If missing: direct the user to install `uv` — `curl -Ls https://astral.sh/uv/install.sh | sh` (macOS/Linux) or `winget install astral-sh.uv` (Windows). `uvx` is bundled with `uv`.

3. **Never hardcode absolute paths** in `mcp.json`. Always use plain `uvx` (not `/Users/<name>/.local/bin/uvx`) so the config is portable.

4. **Never hardcode credentials** in `mcp.json`. Always use `${input:...}` references so secrets are managed by VS Code, not stored in plain text.

## Framework Rules

The rules below are permanent. Do not modify them per-repo — repo-specific context belongs in `.copilot-doctrine.md` and `.agents/rules/`.

### Operational Safety

**Guarded mode — always active by default:**
Any write, mutating, or deployment operation against an external system — cloud infrastructure, managed services, SaaS APIs, databases, messaging systems, or any environment outside localhost — requires explicit user confirmation before execution. This includes provisioning, deploying, updating, or tearing down resources via any tooling (IaC, cloud CLIs, deployment frameworks), as well as any write call to external APIs or platforms.

**Destructive operations — elevated caution:**
Operations that are destructive or irreversible (resource deletion, stack teardown, data removal, force operations) require an extra explicit confirmation beyond the standard approval. Always label these as destructive, state what will be permanently lost or removed, and obtain a distinct second confirmation before proceeding.

**Secrets — never expose:**
Never print, log, suggest committing, or include credentials, API tokens, secrets, or sensitive configuration values in any output, file, or command. Reference secrets by environment variable name or secret manager path only. Flag immediately if a file or context appears to contain secrets that should be rotated or removed.

**Write operations — three-phase protocol:**
Before executing any operation with external side effects (writes, applies, deploys, destroys), follow the three phases below. **Scale depth to impact** — routine low-risk writes get a lightweight pass; high-impact or critical operations get the full treatment.

**Phase 1 — Preflight checks** (always run, depth scales with impact):
- *Lightweight (default)*: confirm target and environment, validate inputs, check for obvious conflicts (resource already exists, wrong account/region, missing permissions).
- *Thorough (High/Critical impact)*: full dry-run / plan / preview (`terraform plan`, `aws cloudformation deploy --no-execute-changeset`, etc.), validate IAM permissions, confirm dependencies and downstream consumers, check for active locks or concurrent changes.

**Phase 2 — Risks and side effects** (always surface before proceeding):
- State what will change and what will not change.
- Name collateral effects on dependent systems, consumers, or data.
- Rate the blast radius: **Low / Medium / High / Critical** — use this scale:
  - *Low*: isolated, easily reversible (e.g. tag update, comment added).
  - *Medium*: limited scope, reversible with manual steps (e.g. config change, new resource).
  - *High*: broad scope or slow/complex rollback (e.g. schema migration, routing change).
  - *Critical*: data loss risk, service disruption, or irreversible (e.g. deletion, destructive deploy).
- For **High / Critical**: state the rollback path and estimated recovery time before requesting approval.

**Phase 3 — Post-validation** (always run after execution, depth scales with impact):
- *Lightweight (default)*: confirm the operation completed without errors; spot-check one key observable (resource exists, endpoint responds, record updated).
- *Thorough (High/Critical impact)*: run health checks or smoke tests, verify dependent systems are unaffected, confirm metrics/logs reflect expected state, update any tracking ticket or session log.

Produce a summary of phases 1–2 and wait for user approval before executing. Show phase 3 results after completion. If preflight reveals failures or unexpected changes, surface them clearly and require re-confirmation before continuing.

### Shell & Code Execution Safety

**Temp file rule:**
Any multi-line code block that will be run or applied must be written to a temp file first. **Never use `/tmp/` or other shared system folders.** Use the active session's `tmp/` folder in `work-sessions` instead:
- If a session is active: `<work-sessions-repo>/sessions/<session-id>/tmp/<task>-<YYYYMMDD-HHmmss>.<ext>` (timestamp prevents collisions)
- If no session is active (e.g. ad-hoc investigation): start one with `#initialize_work_session_folder.prompt.md` first — there is no valid location for scratch files outside a session.

Similarly, any script generated as part of the work must go to `<work-sessions-repo>/sessions/<session-id>/scripts/` — not `/tmp/` or the repo root.

Never pass multi-line content as an inline shell argument or heredoc in a single invocation.

**One command per invocation:**
Run one shell command per invocation. Chain only trivially short, related commands with `&&`. Never use semicolons to chain unrelated commands. Never use subshells (`bash -c "..."`) unless explicitly requested by the user.

**No background processes without notice:**
Always inform the user before starting a background process. Include the terminal ID so output can be checked or the process killed.

**Long-running commands — wait, then report from final output:**
For commands that are still running, never start a duplicate execution, suggest re-running, or infer failure from partial logs. Wait for the final exit status and complete output first. After completion, report results directly from that final output; do not claim truncation or re-run the same command when the terminal already shows a successful exit (e.g. `Exit Code: 0` with relevant success lines). Re-run only if the command failed or essential output is genuinely missing.

### Git Policy

| Operation | Policy |
|---|---|
| `git add`, `git commit` | Autonomous after user approves the commit message |
| `git push` | Always requires explicit user approval |
| `git push --force` | Requires approval + must be labeled as a destructive operation |
| `git worktree add / remove` | Autonomous — describe the action and path before running |
| Reference repo sync (`create-worktree.sh`'s fetch + hard-sync to default branch) | Autonomous — never manual edits to sync, since reference repos never carry local changes |
| PR create / update | Always requires explicit user approval (via `#open-pr.prompt.md`) |
| Session file read/write (in `work-sessions`) | Autonomous |

### Commit Message Convention

All commit messages must follow [Conventional Commits v1.0.0](https://www.conventionalcommits.org/en/v1.0.0/):

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

Allowed types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `perf`, `ci`, `build`, `revert`.

- Description must be lowercase and imperative mood (`add`, `update`, `remove` — not `added`, `updates`).
- Breaking changes must include `BREAKING CHANGE:` in the footer or `!` after the type.
- Always produce a spec-compliant message. Flag and correct any non-compliant message before committing.

### Commit Signing Requirements — Dynamic Discovery

Branch naming conventions and signing rules should be **discovered dynamically at the start of every session** by querying the target repo's GitHub rulesets, rather than hardcoded — rulesets change over time and differ per org/repo.

#### Discovery procedure (run once per session, per target repo)

1. **List all rulesets for the target repo:**
   ```
   gh api repos/<owner>/<repo>/rulesets
   ```
   - If the API returns `403` (private repo on a free plan), fall back to the cached patterns in `/memories/repo/branch-naming.md` if one exists.
   - If the API succeeds, proceed to step 2.

2. **Fetch full details for each ruleset** that has `"target": "branch"`:
   ```
   gh api repos/<owner>/<repo>/rulesets/<id>
   ```

3. **Extract and store the effective rules:**
   - From `conditions.ref_name.include` — branches the rule **applies to**
   - From `conditions.ref_name.exclude` — branches the rule **does not apply to**
   - From `rules[].type` — the enforcement (e.g. `required_signatures`, `branch_name_pattern`)
   - Store results in `/memories/repo/branch-naming.md` for the session

4. **Apply signing rule:** For any ruleset with `"type": "required_signatures"`, use `git commit -S` on every branch that matches the ruleset's `include` patterns (or all branches, if `include` is empty and the branch is not in `exclude`).

Cache what you discover per-org so you don't re-run this every session — but
always re-verify when the cached data is stale or a signing failure suggests
it's out of date.

#### Naming convention

Once you've discovered your org's branch protection rules, document the
resulting naming convention here so it doesn't need rediscovering every
session. A common pattern to adapt:

| Pattern | Purpose |
|---|---|
| `main`, `master`, `develop`, `staging`, `trunk` | Long-lived base branches |
| `release/*` | Release branches |
| `copilot/*` | Agent-created branches |
| `dev/<team>/*` | Team-scoped branches |

#### Ticket key in branch name (mandatory when a ticket is associated)

If the work is linked to a Jira ticket, the ticket key **must** appear in the branch name. Place it immediately after the team/type prefix, before the short description slug:

| Scenario | Branch format | Example |
|---|---|---|
| Agent session with ticket | `copilot/<TICKET-KEY>-<slug>` | `copilot/PROJ-42-route53-iac` |
| Team branch with ticket | `dev/<team>/<TICKET-KEY>-<slug>` | `dev/platform/PROJ-42-route53-iac` |
| No ticket | `copilot/<slug>` or `dev/<team>/<slug>` | `copilot/update-mcp-config` |

- The ticket key must be **uppercase** and match the Jira project key exactly (e.g. `PROJ-123`, not `proj-123`).
- The slug after the key must be lowercase, hyphen-separated, and describe the change concisely.
- If a ticket is discovered mid-session after the branch is already created, note the ticket key in the session file but do **not** rename the branch unless the user explicitly asks.

If commit signing is not configured or signature verification fails, stop and tell the user before attempting to push.

### Model & Context Discipline

A **work session** (the ticket/branch/session-folder in `work-sessions`) is
durable and may span days. The **conversation** you are having with the
agent is not the same thing, and must not be treated as the persistence
layer — file state (`CONTEXT.md`/`TASKS.md`/`PLAN.md`/`SPEC.md`/`WORKLOG.md`,
per `.agents/rules/session-state.instructions.md`) is. One work session
commonly spans **many** conversations.

- **Default the interactive model to Sonnet 5 (or your runtime's equivalent
  mid-tier model).** Switch deliberately to a top-tier model (Opus) for
  planning, architecture, and final review — not as the ambient default for
  routine execution. This mirrors the autonomy tiering in
  `.agents/rules/dev-lifecycle.instructions.md`, applied to interactive work.
- **Watch context growth, not just wall-clock time.** Check your runtime's
  usage/cost indicator periodically (e.g. `/cost` in Claude Code, if
  available). A conversation's cost per turn scales with how much of it gets
  re-sent — that volume is what to manage, not a pricing cliff at any
  specific token count.
- **Flush and start fresh well before a conversation gets large.** As a
  practical ceiling, treat combined context above roughly 100–150K tokens as
  a signal to wrap up: flush state to the session files (see
  `session-state.instructions.md`), then continue in a **new** conversation
  rather than pushing the current one further. Don't let a large one-time
  investigation (a big skill/doc dump, many large file reads, a long
  subagent transcript) linger in a conversation that still has more work
  ahead of it — isolate exploration into a subagent when the finding, not
  the search, is what's needed back in the main thread.
- **Pausing a work session means ending this conversation too.**
  `#pause_work_session.prompt.md` flushes file state; once it's done, close
  the conversation. `#resume_work_session.prompt.md` is designed to run in a
  **new** conversation — it reconstructs enough context from the session
  files, not from a continued transcript. Resuming by continuing the same
  long-running chat defeats this and is the single most common way context
  (and cost) grows unbounded across a break.

### Approval Protocol

Before any guarded operation:
1. Describe exactly what will happen
2. Show the exact command or API call
3. Wait for explicit user approval (`yes`, `approved`, `proceed`, `go ahead` or similar)
4. Never infer approval from context — it must be the most recent user message

### Flex Mode

Activated via `#enable_flex_mode.prompt.md`. Sandbox/localhost autonomous writes are permitted only for the explicitly declared environment. Production, staging, customer-facing environments, and external SaaS APIs remain guarded even in flex mode unless explicitly listed in the flex declaration.
