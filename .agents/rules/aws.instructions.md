## MCP Servers

### aws-api (awslabs.aws-api-mcp-server)

- **Package**: `awslabs.aws-api-mcp-server@latest`
- **Config key in mcp.json**: `aws-api`
- **Inputs required**: `aws-profile`, `aws-region`
- **Tools**: `call_aws`, `suggest_aws_commands`
- **Purpose**: Execute any AWS CLI command via `call_aws`; suggest commands from natural language via `suggest_aws_commands`. Covers all AWS services including those released after model knowledge cutoff.

### aws-docs (awslabs.aws-documentation-mcp-server)

- **Package**: `awslabs.aws-documentation-mcp-server@latest`
- **Runtime**: default `uvx` Python (no credentials required)
- **Config key in mcp.json**: `aws-docs`
- **Inputs required**: none
- **Tools**: `search_documentation`, `read_documentation`, `recommend`
- **Purpose**: Fetch, search, and read public AWS documentation pages

## Tool Priority & Fallback

| Operation | Preferred tool | Fallback |
|---|---|---|
| AWS resource queries / operations | aws-api MCP (`call_aws`) | `aws` CLI |
| AWS documentation lookup | aws-docs MCP | direct HTTPS fetch to docs.aws.amazon.com |

**Fallback trigger**: fall back to CLI/fetch only if the MCP tool is genuinely unavailable. Never fall back on errors — fix parameters and retry.

### Token efficiency — when to prefer CLI over MCP

MCP returns the full API response without filtering. For large datasets this burns context fast. Use CLI with `--query` when the response would otherwise be large:

| Scenario | Prefer |
|---|---|
| Large response, only a subset needed (e.g. listing many resources, filtering by name) | CLI + `--query` |
| Small / targeted response (STS, describe one resource) | MCP |
| Exploratory — unknown shape or size | CLI + `--query` to filter progressively |
| Write operations (create, delete, update) | MCP — safer, no shell parsing risk |

```bash
# Example: fetch only the matching records instead of the full response
aws route53 list-resource-record-sets \
  --hosted-zone-id <zone-id> --profile <profile> \
  --query "ResourceRecordSets[?Name=='example.com.']"
```

## AWS SSO Profiles

Profile names are **personal** — they depend on each engineer's local SSO config and may differ across machines. Do not hardcode profile names in committed files.

**Convention:** pick a consistent naming scheme and document it here, e.g.
`<org>-<account-alias>` (Admin) and `<org>-<account-alias>-ro` (ReadOnly).
**To see your local profiles:** `aws configure list-profiles`
**Prefer `-ro` profiles** for read-only/exploratory operations. Use admin profiles only when writes are required and the user has confirmed.

Record your org's actual values here:
- SSO start URL: `<https://your-org.awsapps.com/start>`
- SSO region: `<region>`
- Default region: `<region>`

Your personal profile-to-account mappings belong in `.agents/rules/aws.local.md` (gitignored). See that file for your aliases.

## Session AWS environment (`.env`)

Every work session folder gets a `.env` (generated by `init-session.sh` from
`session-template/.env.example`; gitignored). It is the single source of truth
for that session's AWS defaults:

| Var | Example | Purpose |
|---|---|---|
| `AWS_PROFILE` | `<org>-test` | Profile Claude, the AWS MCP server, and the CLI target |
| `AWS_DEFAULT_REGION` | `us-east-1` | Region for all AWS calls |
| `AWS_ALLOWED_PROFILES` | `<org>-test,<org>-partner` | Comma-separated allow-list; `aws-login.sh` refuses anything else |
| `CLAUDE_CODE_DONT_INHERIT_ENV` | `true` | Stops Claude Code inheriting stray `AWS_*` from the broader shell |
| `AWS_SSO_BROWSER` | *(unset)* | Optional — browser to route the one-time SSO approval to |

`session-tmux.sh` loads this `.env` into the session's tmux environment on
init/resume, so panes (and Claude launched from them) inherit the right
profile/region automatically. To pick a different profile for a session, edit
its `.env` — do not hardcode profiles elsewhere.

## Reauthenticating profiles

Use `scripts/aws-login.sh` (or the `#aws-reauth` command). It checks
`aws sts get-caller-identity` first and **only** runs `aws sso login` when the
SSO token has actually expired — a valid token is reused with zero browser
interaction, and one login covers every profile sharing the same start URL.

```bash
scripts/aws-login.sh            # reauth the session default profile
scripts/aws-login.sh --all      # every profile in AWS_ALLOWED_PROFILES
scripts/aws-login.sh --list     # show resolved profile/region/allow-list, no auth
```

`aws sso login` needs a TTY/browser — when a login is actually required, run it
via the `!` prompt or in the background, never as a blocking foreground agent
call (it will hang waiting on the browser).

### Gotcha: `#/` in `sso_start_url` forces needless re-logins

AWS CLI caches the SSO token in `~/.aws/sso/cache/` keyed by a hash of the
`sso_start_url`. A trailing `#/` fragment (`…/start#/`) hashes to a *different*
file than `…/start`, so a valid token cached under one is invisible to a
profile configured with the other — every command then reports the session
"expired" and forces a fresh login.

**Fix:** ensure your `sso_start_url` has no `#/` suffix across all profiles.
Check with `grep 'sso_start_url' ~/.aws/config | sort -u`.

### Optional: shared `sso-session` for fewer prompts

Migrating `~/.aws/config` profiles to a shared session block enables refresh
tokens, so the CLI silently re-auths and the browser opens far less often:

```ini
[sso-session org-sso]
sso_start_url = https://your-org.awsapps.com/start
sso_region = us-east-1
sso_registration_scopes = sso:account:access

[profile org-test]
sso_session = org-sso
sso_account_id = <account-id>
sso_role_name = AdministratorAccess
region = us-east-1
```

One `aws sso login --sso-session org-sso` then covers every profile on that
session.

## Operational Scripts — Conventions

### CDK Deploy Guardrails

For CDK apps that derive stack or resource names from `BRANCH_NAME` (or any
other environment-selecting variable), treat that value as a deployment
input, not a local convenience — pin it explicitly rather than relying on
whatever happens to be checked out locally.

**Mandatory preflight before deploy:** run `cdk diff` with the exact same
environment inputs (account, region, app name, env ID, profile, branch/
version selector) you're about to deploy with. If an in-place update is
expected but the diff shows full resource creation, stop and re-verify the
environment inputs before proceeding — a wrong selector value is the most
common cause of an unexpected "create" plan.

### Dry run principle

A dry run that only prints data from a local file gives false confidence — it cannot catch expired credentials, wrong resource IDs, or state that has drifted from a backup.

**A correct dry run must hit the real system read-only** and validate the same inputs the live operation will consume:

- ✅ Fetch from the real API, compare against local/expected state, report MATCH / MISMATCH / NOT FOUND
- ❌ Just print what would happen based on a local file

If the dry run requires credentials and the session is expired, it should fail visibly — that is a feature, not a bug.

### Multi-account / multi-target script reuse

Expose `AWS_PROFILE`, `DRY_RUN`, and similar parameters as environment
variables with sane defaults. This allows the same script to be reused
across accounts without modification:

```bash
DRY_RUN=true AWS_PROFILE=<org>-account-a bash script.sh
DRY_RUN=false AWS_PROFILE=<org>-account-b bash script.sh
```
