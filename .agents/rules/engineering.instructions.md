# Engineering Doctrine

Universal engineering defaults for all work driven from this repo. These are
the baseline; a target repo's `.copilot-doctrine.md` may override them for
that repo (generated by `#set-repo-doctrine.prompt.md`). When a session
starts, surface a one-line reminder of the defaults below alongside the
repo-specific characteristics from `.copilot-doctrine.md`.

## Default Engineering Doctrine

| Principle | Meaning |
|---|---|
| **TDD** | Write a failing test before any implementation. Red → green → refactor. |
| **BDD** | Express acceptance criteria as `Given / When / Then` scenarios before writing code. |
| **Design-first** | Define the interface, contract, or schema before writing implementation. API spec, typed interface, or domain model first. |
| **Well-Architected** | Apply the pillars that are most relevant to this system — default to Operational Excellence, Security, Reliability, Performance, Cost, and Sustainability. Favour managed services, least privilege, and observability. |
| **KIS (Keep It Simple)** | The right amount of complexity is the minimum needed. No speculative abstractions, no over-engineering, no accidental complexity. |

Compact reminder string to show at session start:
`Engineering defaults: TDD · BDD · Design-first · Well-Architected · KIS`

## Testing Doctrine

Approach all implementation work test-first:

- **Red → green → refactor**: write a failing test first, make it pass with minimal code, then refactor.
- **BDD scenarios**: for any feature or acceptance criterion, write `Given / When / Then` scenarios before touching implementation code.
- **Modifying existing behavior**: confirm test coverage exists for the affected paths *before* making changes; if coverage is absent, write the tests first.
- **Agent behavior**: always propose the test(s) and wait for user approval before writing implementation. Never generate implementation code without a prior agreed test or explicit user waiver.

## Design Doctrine

Define the interface, contract, or schema before writing any non-trivial implementation:

- **API/service work**: draft an OpenAPI spec, protobuf definition, or typed interface before handler code.
- **Data model work**: define the domain model or schema before persistence logic or business rules.
- **Agent behavior**: propose the design/interface and wait for user approval before generating implementation. If the design is ambiguous, ask clarifying questions rather than assuming.

## Script Testing Standard

Every non-trivial script in this repo's `scripts/` (and any script a command
shells out to) **must ship with automated tests**, added in the same change as
the script — not later. This is the concrete application of the Testing
Doctrine to tooling.

- **Structure for testability.** Put pure, decision-making logic in a form that
  can be tested without side effects (a Python module, or a bash function with
  no I/O), and keep unavoidable side-effectful orchestration (AWS/SSM, git,
  tmux) in a thin wrapper. Test the pure logic directly; test the wrapper
  end-to-end against throwaway fixtures.
- **No live external systems in tests.** Drive scripts against disposable
  fixtures — temp git repos (bare origin + clone), a scratch session folder, a
  collision-proof tmux session that's always torn down. Never hit real AWS, real
  remotes, the real work-sessions repo, or the user's real tmux sessions. Skip
  (don't fail) a test when a required local tool (e.g. `tmux`) is absent.
- **Runner-agnostic.** Tests are Python `unittest.TestCase` under
  `scripts/**/tests/` — no pytest fixtures — so they run with or without
  pytest. Whole suite: `.venv/bin/pytest scripts`. Dependency-free fallback:
  `python -m unittest discover -s <test-dir>` per directory. A shell script's
  tests drive it via `subprocess` against fixtures (see `scripts/tests/_harness.py`).
- **Cover the contract.** At minimum: the happy path, argument validation /
  `--help`, and every guardrail the script enforces (e.g. refusing a dirty
  source repo, refusing a duplicate session, an incomplete-input gate).
- Existing examples to mirror: `scripts/mongodb/tests/` (pure logic) and
  `scripts/tests/` (end-to-end shell drivers). See `scripts/README.md`.

Trivially exempt: one-off throwaway scripts in a session's `scripts/` folder
(they live in `work-sessions`, not here) and pure documentation.
