## Tool Priority — GitHub Operations

Always prefer GitHub MCP tools over `gh` CLI for any GitHub interaction. Before any operation:
1. Use the MCP tool if it covers the operation
2. Fall back to `gh` CLI **only** if the required MCP tool is genuinely unavailable
3. Never silently fall back — if switching to CLI, state why

> **Common failure mode:** using `gh` for a quick one-off (e.g. `gh pr view`) without loading the MCP first. MCP returns structured JSON directly — fewer tokens, no shell parsing. Always search for the MCP tool first, even for simple reads.

**Never fall back because a tool returned an error.** Fix the parameters and retry the same tool.

### Error handling doctrine (applies to all MCP tools)

| Situation | Action |
|---|---|
| Tool genuinely unavailable | Fall back to next tier |
| Tool returned a parameter/schema error | Fix the parameters, retry the **same tool** |
| Tool returned an API/permission error | Surface full error text to the user, stop — do not silently fall back |
| Persistent failures (3+ retries on same tool) | Surface error to user, ask before falling back or switching strategy |

**Never silently fall back.** If switching tiers, tell the user why and which tool you are using instead.

| Operation | MCP tool (preferred) | CLI fallback |
|---|---|---|
| List / find PRs | `mcp_github_list_pull_requests` | `gh pr list` |
| Create PR | `mcp_github_create_pull_request` | `gh pr create` |
| Update PR | `mcp_github_update_pull_request` | `gh pr edit` |
| Submit review | `mcp_github_pull_request_review_write` | `gh pr review` |
| Read PR + threads | `mcp_github_pull_request_read` | `gh pr view --json` |
| Reply to comment | `mcp_github_add_reply_to_pull_request_comment` | N/A |
| Add issue comment | `mcp_github_add_issue_comment` | `gh issue comment` |
| List commits | `mcp_github_list_commits` | `git log` |

## Commit Message Convention

All commits must follow [Conventional Commits v1.0.0](https://www.conventionalcommits.org/en/v1.0.0/):

```
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

**Allowed types:** `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`, `perf`, `ci`, `build`, `revert`

**Rules:**
- Description must be **lowercase** and **imperative mood** (`add`, `update`, `remove` — not `added`, `removes`)
- Breaking changes: add `!` after the type (`feat!:`) and/or a `BREAKING CHANGE:` footer
- When a Jira/issue ticket is linked to the work, include it in the commit footer:
  ```
  feat(auth): add silent token refresh

  Implements background token renewal to avoid session expiry prompts.

  Refs: PROJ-1234
  ```
- Never hardcode ticket keys in the commit **subject line** — use the footer `Refs:` field instead
- Always produce a spec-compliant message. Flag and correct any non-compliant message before committing.

**Examples:**
```
feat(api): add rate-limit headers to all endpoints
fix(auth): handle null session on token refresh
chore(deps): bump some-dependency to 3.540.0
refactor(sessions): migrate flat-file model to folder-based
docs(readme): update workspace setup instructions

# With ticket reference:
fix(imports): correct mapping for edge case

Refs: PROJ-6424
```

## Branch Naming Convention

| Pattern | Purpose |
|---|---|
| `copilot/<slug>` | Copilot-created branches (default for agent sessions) |
| `copilot/<TICKET-KEY>-<slug>` | Copilot branch linked to a ticket |
| `dev/<team>/<TICKET-KEY>-<slug>` | Team branch linked to a ticket |
| `dev/<team>/<slug>` | Team branch without a ticket |

- **Ticket key must be uppercase** and match the project key exactly (e.g. `PROJ-123`, `ADH-001`)
- The slug after the key must be **lowercase, hyphen-separated**, and describe the change concisely
- If a ticket is discovered mid-session after the branch is already created, note it in the session file but do **not** rename the branch unless explicitly asked

## PR Title Convention

When the PR is linked to a Jira ticket, prefix the title with the ticket key in brackets:

```
[PROJ-6489] Add CloudWatch monitoring for the ingest pipeline
[PROJ-6424] Fix mapping bug for edge-case input
[PROJ-6467] Deny wildcard permissions at org level
```

When there is no linked ticket, use Conventional Commits format:
-Format: `<type>(<scope>): <description>` — same Conventional Commits pattern as commits.

```
feat(sessions): folder-based model with SESSIONS.md tracking
fix(auth): handle null session on token refresh
chore(deps): bump some-dependency to 3.540.0
```

- Ticket key must be **uppercase** and match the Jira project key exactly (e.g. `PROJ-123`)
- Description after the ticket key: sentence case, concise
- Always include the ticket URL in the PR body under **Linked Ticket** as well

## Temp File and Script Placement

When a GitHub operation (PR creation, review submission, etc.) requires writing content to a file before passing it to `gh`:

- **Always** write to the active session's `tmp/` folder: `sessions/<session-name>/tmp/<task>-<YYYYMMDD-HHmmss>.<ext>`
- **Never** use `/tmp/`, `~/.tmp/`, or any shared system folder
- Scripts generated during a session go to `sessions/<session-name>/scripts/` — not `/tmp/` or the repo root
- Use timestamps in filenames to prevent collisions: `YYYYMMDD-HHmmss`

Examples:
```
sessions/PROJ-1234-oauth-refresh/tmp/pr-body-20260413-143022.md
sessions/PROJ-1234-oauth-refresh/tmp/review-456-20260413-151800.md
sessions/PROJ-1234-oauth-refresh/scripts/generate-report.sh
```
